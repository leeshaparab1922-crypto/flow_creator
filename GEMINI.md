# GEMINI.md - CrewAI Guide Creator Flow Instructions & Architecture

This document serves as the primary technical context and development guide for the **Guide Creator Flow** project. It outlines the project's purpose, architectural design, dependency and environment configurations, crew-specific implementations, known discrepancies, and development conventions.

---

## 1. Project Overview & Architecture

### Purpose
The **Guide Creator Flow** is a CrewAI-based flow that automates the transformation of raw educational and learning resources (such as YouTube videos, web documentation, arXiv papers, or local text/PDF/Markdown files) into a high-quality, structured, and beginner-friendly Markdown guide.

### Core Stack
- **Language**: Python (>=3.10, <3.14)
- **Framework**: CrewAI (>=1.15.1, <2.0.0)
- **Dependency & Environment Manager**: `uv` (manifested via `pyproject.toml` and `uv.lock`)
- **Scraping/Automation**: Selenium, Webdriver Manager
- **Validation**: Pydantic (used for Flow State representation)

### Architecture and Flow Steps
The project implements a CrewAI Flow called `GuideGeneratorFlow` defined in `src/flow_defining/main.py` which inherits from `Flow[ResearchFlowState]`.

1. **Input Collection & Validation (`receive_user_input`)**: Accepts and checks user inputs for different resource types. Validates if at least one source type is provided.
2. **Hierarchical Research (`run_research_crew`)**: Spawns the `ResearchCrew` using a hierarchical process. A manager agent coordinates multiple specialized agents to research and compile findings into a comprehensive research report.
3. **Sequential Writing (`run_writing_crew`)**: Spawns the `WritingCrew` using a sequential process. It transforms the comprehensive research report into a beginner-friendly getting-started Markdown guide.

*Note on README.md Discrepancy*: The `README.md` references a three-crew sequence including an **Enrichment Crew** that evaluates research quality and performs additional web searches if needed. However, the current code implementation in `main.py` contains only two active crews (`ResearchCrew` and `WritingCrew`) and skips any intermediate quality scoring or enrichment steps.

## 3. Crews Breakdown & Configurations

### A. Research Crew (`ResearchCrew`)
- **File**: `src/flow_defining/crews/research_crew/research_crew.py`
- **Process**: `Process.hierarchical`
- **Coordinator**: `research_manager` (configured with `allow_delegation: true`)
- **Specialist Agents**:
  - `youtube_specialist`: Uses `YoutubeVideoSearchTool` and `YoutubeChannelSearchTool`
  - `web_specialist`: Uses `ScrapeWebsiteTool`, `SeleniumScrapingTool`, and `CodeDocsSearchTool`
  - `arxiv_specialist`: Uses `ArxivPaperTool` (configured with `download_pdfs=True` and `use_title_as_filename=True`)
  - `document_specialist`: Uses `FileReadTool`, `PDFSearchTool`, `TXTSearchTool`, `MDXSearchTool`, and `DirectoryReadTool`
- **Task**: `research_compilation` (combines findings from all active specialist sources into a minimum 5000-word report structure).

### B. Writing Crew (`WritingCrew`)
- **File**: `src/flow_defining/crews/writing_crew/writing_crew.py`
- **Process**: `Process.sequential`
- **Agents**: `content_writer`, `technical_writer`
- **Tasks**: `review_and_polish_guide` (which outputs to `report.md`) and `write_getting_started_guide`.
- ***Technical Anomaly***: In `WritingCrew.crew()`, the tasks are registered in this order:
  ```python
  tasks=[
      self.review_and_polish_guide(),
      self.write_getting_started_guide()
  ]
  ```
  In sequential execution, `review_and_polish_guide` executes *before* `write_getting_started_guide`. Typically, a writer writes the guide before it is reviewed and polished. Developers should review whether this ordering was intentional or requires refactoring.

### C. Content Crew (`ContentCrew` - Unused)
- **File**: `src/flow_defining/crews/content_crew/content_crew.py`
- **Process**: `Process.sequential`
- **Agents**: `planner`, `writer`, `editor`
- **Tasks**: `planning_task`, `writing_task`, `editing_task`
- **Status**: Currently unused and not imported or utilized in `main.py`.

## 4. Building, Running & Testing

### Environment Setup
The project uses `uv` for python environment and dependency management.

1. **Install uv**: Make sure you have `uv` installed on your machine.
2. **Install Dependencies**:
   ```bash
   uv sync
   ```
3. **Configure Environment Variables**:
   Create a `.env` file at the root of the project with the following keys:
   ```env
   OPENAI_API_KEY=your_openai_api_key
   SERPER_API_KEY=your_serper_api_key
   ```

### Execution Commands
You can run the application using either the direct `uv run` scripts defined in `pyproject.toml` or the standard `crewai` CLI.

#### Using `uv run`
- **Run the complete Flow (terminal interactive)**:
  ```bash
  uv run kickoff
  ```
  *(or `uv run run_crew`)*

#### Using `crewai` CLI
- **Kickoff the Flow**:
  ```bash
  crewai flow kickoff
  ```

#### Anomaly in Script Entrypoints (`pyproject.toml`)
The `pyproject.toml` file declares two scripts that are currently missing implementation in `src/flow_defining/main.py`:
- `plot = "flow_defining.main:plot"` (Fails because `plot()` is not defined in `main.py`)
- `run_with_trigger = "flow_defining.main:run_with_trigger"` (Fails because `run_with_trigger()` is not defined in `main.py`)

To plot the flow visually, you should instead use the official CrewAI CLI:
```bash
crewai flow plot
```

### Testing
- The `tests/` directory is currently empty.
- To set up testing, add `pytest` or your preferred testing framework, write your test modules in the `tests/` directory, and run them using `uv run pytest` (once installed).

## 5. Development Conventions

When extending or maintaining this codebase, adhere to the following conventions:

1. **Adding Crews**:
   - Create a subfolder under `src/flow_defining/crews/` named `<new_crew_name>_crew`.
   - Maintain the separation of concerns: place YAML config files under `config/agents.yaml` and `config/tasks.yaml`.
   - Implement the crew definition class utilizing `@CrewBase`, `@agent`, `@task`, and `@crew` decorators from `crewai.project`.

2. **Developing Tools**:
   - Custom tools should be defined under `src/flow_defining/tools/` using the subclassing pattern shown in `custom_tool.py`.
   - Ensure you declare type-safe input schemas using Pydantic `BaseModel`.

3. **Writing Conventions**:
   - Ensure rigorous type-hinting across all function signatures.
   - Flow states must be represented using Pydantic `BaseModel` models and declared on the `Flow[StateClass]` generic parameter.
   - Do not commit `.env` files or hardcode API credentials. Protect system configuration and private setup folders.
